# 여행 추천 Agent 개발 계획서 (OpenAI 일정 생성 위임)

## 1) 목표 및 범위

### 목표
- 사용자 입력(`성별, 성격, MBTI, 취향, 나이, 경비`) + 선택한 `국가/주요 도시`를 바탕으로
- 각 도시별 **주요 여행지 추천(관광/식사/이동 동선/시간대)** 및 **여행 계획(일정표)**을 생성하는 Agent를 개발

### 이번 버전의 핵심 결정
- **여행 일정(관광/식사 등) “추천 생성”은 OpenAI이 수행**하도록 설계
- 우리의 Agent는 OpenAI에게 요청하기 전에:
  - 사용자 입력을 제약/가중치로 변환
  - 도시별 후보군(관광지/식당 등)을 구성해 “환각 방지”를 강화
  - 출력 스키마 검증/예산·일정 제약 체크 및 필요 시 재생성(리플랜)
  - 결과를 사용자 친화 포맷으로 정리

### 포함 범위
- 입력 정규화(User Profile)
- 도시/권역 기반 후보군 구성
- OpenAI 프롬프트/출력 스키마 설계
- JSON 스키마 검증 및 예산·과밀 일정 방어

### 제외/후순위
- 실시간 길찾기/교통 API 완전 통합(초기에는 추정치로 대체)
- 모든 도시/식당의 완전한 DB 커버리지(초기에는 MVP 도시 중심)

## 2) 입력/출력 스펙

## 2.1 입력 스키마(사용자 → Agent)
- `gender` (선택): `여/남/기타/선택안함`
- `personality` (선택):
  - 기본 선택: `외향/내향/차분함/활동적/감성적/...`
  - 자유서술(있으면): 보조 신호
- `mbti` (선택): `ENFP/ISTJ/...`
- `preferences` (권장): 다중 선택(가중치 부여 가능)
  - 예: `자연, 카페, 쇼핑, 전시, 박물관, 야경, 액티비티, 로컬맛집, 미술, 테마파크, 해변`
- `age` (선택): 숫자 또는 `10대/20대/...`
- `budget` (권장):
  - 모드 A: `저/중/고` (1인 기준 1일 예산 범주)
  - 모드 B: 총액 또는 1일 금액(가능하면)
- `trip_days` (권장): 예) `2박3일/3박4일` 또는 `N일`
- `countries` (선택): 국가 리스트
- `cities` (선택): 국가별 주요 도시 복수 선택
- `travel_style` (옵션): `가족/커플/혼자/친구` 등

## 2.2 출력 스키마(OpenAI 결과 → Agent 검증/사용자 표시)
> 초기에는 **JSON 스키마 고정**을 강제해 품질을 안정화한다.

- `summary`
  - 일정 전체 요약(짧게)
- `profile_basis`
  - 어떤 입력(성향/취향/MBTI/나이/예산)이 왜 반영됐는지 간단 근거
- `itineraries`
  - 도시 단위 → 날짜 단위 → 시간대 단위 항목 배열
  - 각 항목:
    - `time_slot`: 예) `09:30-11:30`
    - `type`: `관광/식사/카페/이동/휴식/액티비티`
    - `name`: 장소명(후보군 기반)
    - `area`: 권역/동네 태그
    - `why`: 추천 이유(짧게)
    - `estimated_cost_range`: 예) `저/중/고` 또는 `대략 10,000~20,000원`
    - `duration`: 분 단위 소요(선택)
- `food_plan`
  - 아침/점심/저녁(또는 브레이크타임) 추천 구조
- `transport_notes`
  - 이동 동선 팁(권역 묶음 중심)
- `budget_check`
  - 예산 충족/근소 초과/초과 및 조정안(규칙 기반)
- `alternatives`
  - 비 오는 날/덜 걷기/쇼핑 제외/카페 더 등 대안 플랜 2~3개

## 3) Agent 설계(역할 분리)

### 3.1 Agent 워크플로우(추천 생성 요청 전/후)
1. **입력 정규화(User Profile Normalization)**
   - 성격/취향/MBTI를 내부 표준 태그로 매핑
   - 선호 가중치 계산(예: `내향+감성적`이면 카페/전시/야간 산책 가중치↑)
2. **Itinerary Blueprint(뼈대) 생성**
   - 하루 템플릿 고정:
     - 오전 관광 → 점심 식사 → 오후 관광/전시 → 휴식/카페 → 저녁 식사 → 야경(가능 시)
   - 체력/나이/성격에 따라:
     - 걷기 강도(이동/대기 빈도)
     - 관광지 수(하루 최대치)
3. **City Candidate Retrieval(후보군 구성)**
   - 도시별 관광지/식당/체험 후보를 DB/데이터셋에서 가져와 OpenAI에 함께 제공
   - 핵심: **OpenAI이 “없는 장소를 지어내지 못하도록” 후보군을 함께 제시**
4. **OpenAI Generation(일정 생성 요청)**
   - 프롬프트에:
     - 사용자 프로필 요약
     - 후보군 리스트(장소명/카테고리/권역/가격대/소요시간)
     - 제약(시간대 구조, 후보군 외 장소명 금지, 예산/과밀 방지)을 명시
   - JSON 스키마로 출력 강제
5. **Post-processing & Validation**
   - JSON 파싱 및 필수 필드 체크
   - 장소명/ID가 후보군과 일치하는지 검증(매칭 실패 시 재생성)
   - 예산 계산:
     - 관광/식사/입장료/교통 추정치 합산
   - 과밀 일정 검증:
     - 하루 관광지 수 상한, 이동 횟수 상한, 휴식 블록 포함 여부
   - 위반 시:
     - 해당 슬롯만 저가/무료 대안 후보로 교체 후 재생성(또는 부분 수정)

## 4) 데이터/후보군 설계(환각과 품질의 핵심)

### 4.1 권장 최소 필드

#### 관광지 후보 테이블(필수)
- `id`
- `city`
- `name`
- `category` (관광/전시/야경/자연/쇼핑/체험 등)
- `area_tag` (권역: 구시가지/강변/역세권/박물관거리/쇼핑가 등)
- `duration_minutes`
- `price_tier` (저/중/고 또는 무료/유료)
- `weather_suitability`(선택)

#### 식당 후보 테이블(필수)
- `id`
- `city`
- `name`
- `cuisine_tags` (로컬/일식/양식/한식/카페 디저트 등)
- `area_tag`
- `meal_type` (breakfast/lunch/dinner)
- `price_tier`
- `localness`(선택)

#### 액티비티 후보(선택)
- `id`
- `category=activity`
- `duration_minutes`
- `price_tier`
- `time_suitability` (오전/오후/야간)

### 4.2 MVP 데이터 전략
- 도시 커버리지는 작게 시작(예: 5~10개 도시)
- 대신:
  - 권역 태깅을 잘하고
  - 카테고리/가격대 라벨링을 일관되게 운영
- 이렇게 해도 일정 품질이 크게 좋아짐(동선 묶음이 자연스러워짐)

## 5) OpenAI 프롬프트 전략(생성 품질을 좌우)

### 5.1 프롬프트 원칙
- **후보군을 함께 제공**
- “후보군 목록 밖 장소명은 절대 사용 금지”를 명시
- 출력은 **스키마 고정 JSON**으로 강제
- 제약(하루 관광지 수/휴식 포함/예산 범위)을 룰로 명시

### 5.2 추천 요청 흐름(템플릿 관점)
1. 시스템/정책:
   - 일정 생성기
   - 환각 방지: 후보군 외 장소명 금지
   - JSON 스키마 준수
2. 사용자 메시지:
   - 프로필 요약(성향/취향/나이/예산)
   - 도시/권역 후보군 리스트(관광/식당/체험)
   - 여행 기간 및 템플릿
   - 출력 JSON 스키마

## 6) 검증/품질 보장 장치

### 6.1 환각(없는 장소 생성) 방지
- OpenAI에게 “후보군 외 장소명 금지”를 명시
- 결과 JSON에서:
  - `itineraries[].name`이 후보군 `id/name` 매칭에 실패하면 재생성

### 6.2 예산 실패 방지
- 후보군에 `price_tier` 또는 대략 비용 범위를 부여
- Agent에서:
  - 하루/총 예상 비용 합산
  - 초과 발생 시:
    - 저가/무료 후보로 교체하여 재생성

### 6.3 과밀 일정 방지
- 하루 최대 관광 슬롯 수 제한(예: 4개 내)
- 이동 횟수 상한
- 휴식/카페 블록 최소 포함
- 위반이면 재생성(또는 해당 구간만 축소)

## 7) 사용자 UX(생성 후 수정/대안)

- 생성 결과에서 스위치 형태로 재생성:
  - `더 걷기 싫어요`
  - `덜 비싸게`
  - `비 오는 날로 바꿔줘`
  - `쇼핑 제외`
  - `카페/전시 더`
  - `액티비티 비중↑/↓`
- 피드백 수집:
  - 각 결과에 “좋아요/별로” 또는 수정 요청 항목을 저장
- 다음 추천에 가중치 튜닝 반영

## 8) 마일스톤(권장 4단계)

### 1단계: MVP(1~2주)
- 입력 폼 + 국가/도시 선택
- 1개 도시 또는 고정된 기간(예: 3일)에서
  - OpenAI로 일정 생성
- 결과는 최소 스키마로만 출력

### 2단계: 후보군 기반 환각 방지(1~2주)
- 도시별 후보군 데이터셋 구축 최소 버전
- OpenAI 입력에 후보군 포함 + 출력 검증

### 3단계: 제약 엔진 강화(1~2주)
- 나이/체력/성격 기반:
  - 하루 관광 수 상한
  - 휴식 삽입 규칙
  - 예산 상한 규칙
- 초과/위반 시 교체 후보 재생성

### 4단계: 대안/피드백(1~2주)
- 비/체력/취향 변화에 따른 대안 플랜 생성
- 사용자 좋아요/수정 로그로 가중치 튜닝

## 9) 다음 결정이 필요한 3가지(진행 속도에 영향)
1. MVP에서 여행 기간을 고정할까요?(`2박3일` vs `3박4일` 등)
2. 예산 입력을 `저/중/고` 범주형으로 시작할까요, 숫자 금액으로 시작할까요?
3. OpenAI에 전달할 후보군 데이터 범위:
   - 관광지만 먼저 할지
   - 식당까지 동시에 할지

- 출력은 반드시 JSON 스키마로 고정
- 결과 검증 실패 시 자동 재생성 루프
- 예산/과밀 위반 시 “슬롯 교체” 방식으로 부분 재생성
